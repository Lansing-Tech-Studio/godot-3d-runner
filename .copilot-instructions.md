# Godot 3D Runner - Project Instructions

## Project Overview
A minimal third-person runner demo in Godot 4.6. The player runs on a large flat arena bounded by distant colored walls, with a grid floor for motion reference. Focus is on responsive movement, smooth camera follow, and clean code rather than complexity.

**Target**: Fast, responsive gameplay feel with minimal visual noise and good frame performance on moderate hardware.

## Architecture

### Entry Point
- **Main scene**: `res://scenes/main.tscn`
- Root node: `Node3D` with world setup script
- Child: `Player` CharacterBody3D with movement and camera scripts

### Core Scripts

#### `scripts/player_controller.gd`
- **Extends**: `CharacterBody3D`
- **Responsibilities**:
  - Input handling (move_forward, move_backward, turn_left, turn_right, sprint)
  - Velocity and acceleration management
  - Gravity and floor detection
  - Camera rig creation and positioning
  - Visual mesh (capsule) for the character
  - Collision shape setup

- **Key exports**:
  - `walk_speed` (default 8.0): base forward speed
  - `reverse_speed` (default 4.5): backward speed cap
  - `sprint_max_speed` (default 48.0): Shift+W acceleration cap
  - `speed_change_rate` (default 10.0): normal acceleration
  - `reverse_deceleration_rate` (default 20.0): faster decel when reversing direction
  - `turn_speed` (default 2.5): degrees per second rotation
  - `gravity_scale` (default 1.0): multiplier on world gravity

- **Behavior**:
  - W moves forward (relative to current facing)
  - S moves backward at reverse_speed
  - A rotates counterclockwise, D rotates clockwise
  - Shift+W accelerates smoothly to sprint_max_speed
  - Releasing W and pressing S applies fast reverse deceleration for snappy direction changes
  - Camera pivot is offset above and behind the player, rotates with player heading
  - No head bob or secondary camera motion (intentionally simple)

#### `scripts/world_setup.gd`
- **Extends**: `Node3D` (root of main scene)
- **Responsibilities**:
  - Environment setup (DirectionalLight, sky, atmosphere)
  - Floor mesh and collision
  - Floor grid markers (visual reference for motion)
  - Four boundary walls at cardinal directions

- **Key constants**:
  - `ARENA_HALF_EXTENT` (200.0): distance from origin to arena edge
  - `WALL_HEIGHT` (18.0): wall mesh height
  - `WALL_THICKNESS` (2.0): wall depth
  - `GRID_SPACING` (10.0): distance between grid lines
  - `WALL_COLORS`: array of 7 colors cycling through walls

- **Behavior**:
  - Runs once at startup via `_ready()`
  - Each setup method checks if the node already exists before creating (idempotent)
  - Materials are pooled and reused via `_get_material()` and `_get_grid_material()` to reduce memory
  - Floor is a large flat plane; walls are solid colored boxes
  - Grid lines use alternating bright/dim colors every 5 lines for visual rhythm
  - Player spawns at origin (0, 0, 0) height ~1.2 units above floor

## Input Map (Godot Project Settings)
Defined in `project.godot` under `[input]`:
- `move_forward`: W key
- `move_backward`: S key
- `turn_left`: A key
- `turn_right`: D key
- `sprint`: Shift key

All can be remapped via Input Map in Godot settings.

## Physics & Collisions
- **Physics Engine**: Jolt Physics (set in project.godot)
- **Player**: `CharacterBody3D` with `CapsuleShape3D` collision (radius 0.45, height 1.2)
- **Floor**: `StaticBody3D` with large box collision underneath
- **Walls**: `StaticBody3D` with box collisions (no trigger zones; walls are solid)

## Performance Considerations

### Current Optimizations
1. **Material Pooling**: Materials are cached by color; each unique color reuses the same material instance instead of creating duplicates
2. **Static World**: Floor and walls are `StaticBody3D`, so physics engine doesn't re-calculate their motion
3. **Simple Visuals**: Single capsule for player, flat meshes for floor/walls; no particle effects, no animation system
4. **Grid Efficiency**: Floor grid is drawn with simple box meshes, one per line, no overdraw

### Known Limitations
- Grid generation is O(n) on arena size; very large arenas (500+ extent) may have slight startup overhead
- No occlusion culling; walls are always rendered even if far away
- No LOD (level of detail) system for distant objects

### Tuning for Performance
If frame rate drops:
1. Increase `GRID_SPACING` (10.0 → 20.0 reduces grid line count by 50%)
2. Reduce `ARENA_HALF_EXTENT` to shrink the playable area
3. Reduce shadow quality on `DirectionalLight3D` (set `shadow_enabled = false` if needed)
4. Lower camera FOV slightly (75 → 60) to reduce rendered geometry

## Movement Mechanics
- **Forward motion**: Velocity is computed in world space relative to player's current facing direction (−global_transform.basis.z)
- **Acceleration curve**: Uses `move_toward()` with configurable `speed_change_rate`, applies instantly to target then ramps
- **Reverse deceleration**: When switching from forward to backward (or vice versa), uses `reverse_deceleration_rate` for snappy feel
- **Gravity**: Applied every frame unless `is_on_floor()` returns true; when on floor, clamps velocity.y ≥ 0 to prevent falling through
- **Camera follow**: Camera is a child of a pivot node that rotates with the player; pivot is positioned ahead and above for third-person offset

## Project Conventions
1. **Node naming**: Use PascalCase for node names (e.g., `Player`, `FloorMarkers`, `CameraPivot`)
2. **Script naming**: Use snake_case for file names (e.g., `player_controller.gd`, `world_setup.gd`)
3. **Constants**: All caps with underscores (e.g., `ARENA_HALF_EXTENT`, `GRID_SPACING`)
4. **Private variables**: Prefix with `_` (e.g., `_forward_speed`, `_gravity`)
5. **Exports**: Use `@export` for tunable parameters so designers can adjust via Inspector
6. **Code organization**: Ensure methods are idempotent (`_ensure_*` pattern) so they can be called multiple times safely

## Common Modifications

### Changing movement speed
- Edit `walk_speed`, `reverse_speed`, `sprint_max_speed` in player_controller.gd exports or Inspector
- Adjust `speed_change_rate` and `reverse_deceleration_rate` to tune acceleration feel

### Changing arena size
- Edit `ARENA_HALF_EXTENT` in world_setup.gd (e.g., 200.0 → 300.0 for larger arena)
- Walls and floor scale automatically

### Changing wall colors
- Edit `WALL_COLORS` array in world_setup.gd or modify `_get_material()` caching
- Each wall cycles through colors by index passed to `_create_rainbow_wall()`

### Adding visual polish
- Player mesh can be swapped in `_ensure_visual()` (currently CapsuleMesh)
- Camera height/distance: edit `CameraPivot.position` and `Camera3D.position` in `_ensure_camera()`
- Lighting: adjust DirectionalLight rotation, energy, and shadows in `_ensure_environment()`

### Testing new mechanics
- Keep `_ready()` idempotent so changes can be tested without reloading the scene
- Use `_physics_process()` for movement, not `_process()`, to stay synchronized with physics
- Verify gravity is applied correctly: test jumping or walking off an edge (if added)

## Future Enhancement Ideas

2. **Character model**: Replace capsule with imported humanoid model
3. **Sprint UI**: Add on-screen indicator for sprint stamina or current speed
4. **Advanced camera**: Add spring arm, bobbing, or look-ahead based on movement direction
5. **Level variety**: Create multiple arenas with different themes
6. **Collectibles**: Add pickup targets to test seek/acquisition gameplay
7. **Audio**: Footstep sounds, wind, impact sounds
8. **Multiplayer**: Sync player position and input across network

## Notes for Future Dev
- This is a deliberate minimal scope. Resist feature creep; keep focus on responsive feel.
- Material pooling is effective; if adding new visual elements, follow the same pattern.
- Physics feel (acceleration, turn speed, gravity) is tunable; small adjustments have large impact on player experience.
- The MCP addon in `addons/godot_mcp/` is for editor integration only and does not affect runtime.
